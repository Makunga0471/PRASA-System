# Reflection

The process of translating system requirements into use case diagrams, detailed use case specifications, activity workflows, and object state models for the PRASA Smart Digital Monitoring System proved to be both intellectually demanding and highly informative. Across these assignments, a key challenge was maintaining consistency and alignment between functional requirements, user stories, use cases, and system models. Ensuring that each diagram and model accurately reflected the intended system behavior required careful interpretation and continuous cross-referencing with previously defined requirements.

One of the primary challenges encountered was determining the appropriate level of detail in both use case specifications and system models. Initially, there was uncertainty regarding how extensively to elaborate basic and alternative flows in use cases, as well as how many states and transitions should be included in state diagrams. It became evident that clarity and precision were essential, but excessive detail could make diagrams complex and difficult to interpret. Striking a balance between completeness and readability was therefore critical throughout the modeling process.

The inclusion of alternative flows and exceptional scenarios, such as sensor malfunctions and network failures, significantly enhanced the robustness of the system design. These scenarios provided deeper insight into how the system should behave under abnormal conditions, ensuring that the PRASA monitoring system is resilient and capable of handling real-world operational challenges. Similarly, state transition diagrams further reinforced this understanding by illustrating how system objects—such as trains, alerts, and maintenance requests—evolve over time in response to events.

The development of activity diagrams introduced another layer of complexity, as it required modeling end-to-end workflows with clear sequencing, decision points, and interactions between different actors. Representing processes such as alert generation, maintenance workflows, and user authentication highlighted the importance of logical flow and system coordination. Differentiating between sequential and parallel processes also emphasized how real-time systems must efficiently handle multiple operations simultaneously.

Identifying and categorizing system actors, including control room operators, maintenance technicians, system administrators, and IoT sensors, was essential for accurately modeling system interactions. The use of relationships such as «include» and «extend» in earlier use case diagrams, combined with workflow and state modeling in later assignments, improved the structural clarity of the system and provided a more comprehensive representation of its functional and dynamic behavior.

The formulation of test cases and traceability mapping further strengthened the understanding of system requirements. Linking each test case, diagram, and workflow to specific functional requirements and user stories ensured that all aspects of the system were verifiable and aligned with stakeholder needs. This process highlighted the importance of defining measurable acceptance criteria, particularly for non-functional requirements such as performance, security, and responsiveness.

Assignment 7 introduced the practical application of Agile project management using GitHub Projects, which added another dimension to the development process. Selecting and customizing the Kanban board required thoughtful consideration to ensure that it accurately represented the workflow of the PRASA system. Challenges included defining appropriate workflow stages, linking user stories and requirements to tasks, and maintaining consistency across the project. This experience reinforced the importance of structured task management and continuous progress tracking in Agile environments.

Comparing GitHub Projects with tools such as Trello and Jira provided valuable insights into different approaches to project management. While GitHub Projects offers strong integration with code repositories, making it ideal for software development, tools like Trello provide a more intuitive interface, and Jira offers advanced features for large-scale enterprise projects. This comparison highlighted the need to select tools based on project complexity and team requirements.

A key learning outcome from Assignment 8 was understanding the distinction between state diagrams and activity diagrams. State diagrams focus on how system objects change over time, representing different states and transitions triggered by events. In contrast, activity diagrams focus on process flows, illustrating the sequence of actions, decisions, and interactions within the system. Together, these models provide complementary perspectives that are essential for designing complex systems.

Overall, these assignments provided valuable practical insight into both software engineering and Agile project management principles. They demonstrated how theoretical requirements can be systematically transformed into structured models, validated through test cases, and managed using Agile workflows. The importance of maintaining accuracy, clarity, and consistency throughout the system development lifecycle was strongly emphasized.

In conclusion, the knowledge and skills gained from these assignments will be highly beneficial in future software development projects. They have strengthened competencies in system analysis, modeling, testing, and Agile project management, while reinforcing the importance of a methodical and detail-oriented approach to developing reliable, efficient, and scalable software systems.
